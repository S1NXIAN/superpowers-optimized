# Cross-Session Memory System — Design Spec

**Date:** 2026-05-29
**Status:** Draft for review
**Author:** Zeus orchestrator

---

## Memory Root

All memory files (project map, context snapshot, known issues) live in
`<project-root>/zeus/memory/` — a **per-project** directory. Each project gets
its own isolated memory silo. Zeus creates `zeus/memory/` silently on first
session init if it doesn't exist.

## 1. Scope & Non-Goals

### In scope
- Four-component cross-session memory stack for opencode-zeus
- Allow-list registration for existing projects at setup time
- Staleness detection and automatic partial refresh
- Graceful degradation on non-git projects
- Testability of every component in isolation

### Non-goals
- Real-time tool-interception hooks (not possible without OpenCode plugin API)
- Cross-session memory for arbitrary JSON state (limited to the four defined components)
- Automatic context window management (the AI manages its own context)
- Multi-repo monorepo support (single `.git` root assumed)

## 2. Components

### 2.1 Project Map (`zeus/memory/project-map.md`)

**Purpose:** Eliminate re-exploration. The AI reads this instead of globbing and skimming the project on every session.

**Location:** Per-project at `<project-root>/zeus/memory/project-map.md`.

**Mechanism:**
1. Generated by `lib/project-map.mjs` — run once per project (user says "map this project" or zeus offers on first build intent)
2. Zeus reads it from `<root>/zeus/memory/project-map.md` at session start — loaded manually, not via instructions array
3. Hard cap: **100 lines** when injected into context. Beyond that, truncate to Hot Files + Critical Constraints sections with a note: "(Full map: <path>)"
4. Git hash in header (`Git: a4b9c2d`) — zeus compares to HEAD at session start
5. Stale? → `git diff --name-only <hash> HEAD` → re-read only changed files → update entries inline

**Graceful degradation:**
- No git: no staleness detection. Map still loads and works.
- No map file: zeus offers to generate one on first build-intent task. User can decline.
- Map >100 lines: zeus truncates in-context version, full version at `<root>/zeus/memory/project-map.md` for reference reads.

**Format:**
```markdown
# Project Map
Generated: 2026-05-28 | Git: a4b9c2d

## Directory Structure
<2-5 lines — tree layout of src/ modules>

## Key Files
<5-15 entries — path + 1-line description of each>

## Critical Constraints
<3-5 rules — non-negotiable project conventions>

## Hot Files
<3-5 paths — most frequently modified files>
```

### 2.2 Context Snapshot (`zeus/memory/context-snapshot.json`)

**Purpose:** Eliminate git command overhead. The AI knows exactly what changed since last session without running `git diff` or `git log`.

**Location:** Per-project at `<project-root>/zeus/memory/context-snapshot.json`.

**Mechanism (two layers):**

**Layer 1 — Git post-checkout hook (passive):**
- Installed by `bin/setup.mjs` as `.git/hooks/post-checkout`
- Runs on every `git checkout` / branch switch
- Builds and writes `zeus/memory/context-snapshot.json` at project root (creates `zeus/memory/` if needed)
- Writes: git hash, changed files (diff HEAD~1..HEAD), change stat, recent 5 commits, blast radius

**Layer 2 — Zeus fallback (active, validated):**
- If `zeus/memory/context-snapshot.json` is missing or `git_hash` != `HEAD`:
  1. Run `git diff HEAD~1..HEAD` → get changed files
  2. Run `git grep --files-with-matches` per changed file → get blast radius
  3. Write `zeus/memory/context-snapshot.json`
- **Validation:** zeus parses the JSON and checks for required keys (`git_hash`, `changed_files`, `blast_radius`) before using. Invalid JSON → rebuild.

**Graceful degradation:**
- No `zeus/memory/` dir: Zeus creates it silently on first session init.
- No git: skip snapshot entirely. Zeus proceeds without it.
- No hook: zeus fallback handles it (adds ~1 tool call overhead).
- No `rg` for blast radius: fallback to native `grep` with Jaccard-like file patterns. If both unavailable, skip blast radius and populate other fields.
- Snapshot stale (hash mismatch): zeus rebuilds.

**Format:**
```json
{
  "git_hash": "a4b9c2d",
  "changed_files": ["src/auth/middleware.js"],
  "change_stat": "1 file changed, 8 insertions(+)",
  "recent_commits": ["a4b9c2d Fix token expiry"],
  "blast_radius": {
    "src/auth/middleware.js": ["src/routes/login.js", "src/tests/auth.test.js"]
  },
  "generated": "2026-05-29T10:00:00.000Z"
}
```

### 2.3 Known-Issues Database (`zeus/memory/known-issues.md`)

**Purpose:** Eliminate re-debugging. The AI checks this before investigating any error.

**Location:** Per-project at `<project-root>/zeus/memory/known-issues.md`.

**Mechanism:**
1. Maintained at `<project-root>/zeus/memory/known-issues.md`
2. Each entry: error signature (code and message), root cause, fix, first-seen date
3. `lib/known-issues.mjs` — reads file, returns entries matching a given error string
4. **Fuzzy matching** using Jaccard similarity on non-stop-word tokens between the error message and each entry's title line. Threshold >= 0.3 yields a match. Lower thresholds capture more but with more false positives.
5. Zeus reads `known-issues.md` at session start if it exists
6. `error-recovery` skill appends entries when a recurring error is diagnosed

**Graceful degradation:**
- No file: skip. Zeus proceeds without it.
- Malformed entry: skip that entry, log warning, proceed. Don't fail.
- File >200 entries: log warning that lookup may be slow. Matching still works.

**Format:**
```markdown
## Cannot read properties of undefined (reading 'name')

**Error:** `TypeError: Cannot read properties of undefined (reading 'name')` at src/routes/users.js:42
**Root cause:** User object destructured before optional chaining
**Fix:** Add optional chaining: `user?.name ?? 'Guest'`
**Context:** Only triggers when user record has null profile field
**First seen:** 2026-05-22

## Database timeout

**Error:** `SequelizeConnectionError: connect ETIMEDOUT` or `pool.query` timeout
**Root cause:** PostgreSQL pool max_connections too low (5) for concurrent workload
**Fix:** Increase pool max to 20 in src/lib/db.js:12
**Context:** Affects production deploy only; local dev uses SQLite
**First seen:** 2026-05-20
```

### 2.4 Command Guard (Agent-Level)

**Purpose:** Prevent catastrophic mistakes. The AI self-screens Bash commands before execution.

**Mechanism:**
1. `lib/command-guard.mjs` — exports a list of dangerous command patterns with 3-tier severity (CRITICAL, DANGEROUS, SUSPICIOUS)
2. Referenced by `agent/zeus.md` in a new "Pre-Execution Safety" section in Core Constraints
3. The agent instructions tell Zeus to check every Bash command against the pattern list before running it
4. CRITICAL matches require override prefix (`DANGEROUS_CMD_ACCEPTED=true`)
5. Not enforced at platform level — advisory but with explicit protocol

**Graceful degradation:**
- Always available (it's a code module + agent instruction, unconditional).
- AI can ignore the guard — but that's a violation of instructions, which is detectable by the security-reviewer agent.

**Pattern tiers:**
```mjs
CRITICAL: [rm -rf /, rm -rf $HOME, git push --force, mkfs, dd, chmod -R 777 /]
DANGEROUS: [> file truncation, drop table/database, kill -9, chmod -R sensitive dirs]
SUSPICIOUS: [rm .git internals, chown, sudo on unusual commands]
```

## 3. Session Init Sequence (Zeus)

All memory files live in `<project-root>/zeus/memory/`. Zeus creates this directory
silently on first session init if it doesn't exist.

```
## Session Init (before classification)

mkdir -p zeus/memory

1. Read zeus/memory/context-snapshot.json (if exists):
   - Validate JSON has required keys (git_hash, changed_files, blast_radius)
   - If stale (hash != HEAD) or invalid: rebuild via git commands
   - Output: `[Session: <X> files changed since last session. Blast radius: <paths>]`

2. Check zeus/memory/project-map.md staleness:
   - Read git hash from map header
   - If hash != HEAD: git diff --name-only <hash> HEAD → re-read only changed files
   - Update map entries inline

3. Check zeus/memory/known-issues.md:
   - If exists: read entries into working memory
   - When systematic-debugging skill fires: check known issues first before investigation
   - If not: no action

4. Command guard pre-flight:
   - Before every Bash command: match against lib/command-guard.mjs patterns
   - If CRITICAL match: require DANGEROUS_CMD_ACCEPTED=true prefix
   - If DANGEROUS match: log warning before executing
```

## 5. Testing Strategy

| Component | Test approach | Key assertions |
|---|---|---|
| `project-map.mjs` | Temp dir with fake git repo | Staleness detection: hash match → not stale; hash mismatch → stale. Generation: produces well-formed markdown. |
| `context-snapshot.mjs` | Temp dir with git commits | Build: correct changed_files, blast_radius. Validate: valid JSON → passes; missing keys → fails. |
| `known-issues.mjs` | Fixture known-issues.md with 5 entries | Fuzzy match finds correct entry. No match returns empty. Malformed entry skips gracefully. |
| `command-guard.mjs` | Pattern list as exported data | Each tier has ≥1 pattern. All patterns are valid regex. Critical patterns match known dangerous commands. |

1. Build each lib module + tests (TDD, RED→GREEN→REFACTOR)
2. Modify `agent/zeus.md` to add Session Init sequence
3. Modify `opencode.json` instructions
4. Build `scripts/post-checkout-hook.sh`
5. Modify `bin/setup.mjs` to install hook + new files
6. Run full test suite
7. Manual test: fresh install on new project, verify all four components work

## 7. Failure Mode Check

| Failure mode | Severity | Mitigation |
|---|---|---|
| Known-issues.md never created by user | **Minor** — degrades gracefully (skip), no crash | Documented non-goal; user must invoke error-recovery skill |
| Git hook fails on Windows | **Minor** — zeus fallback handles it | Fallback rebuilds snapshot on demand |
| Context-snapshot.json grows stale during long session | **Minor** — generated once at start, may drift over multi-hour session | Session-level staleness not tracked; acceptable limitation |
| Command guard gives false sense of security | **Minor** — AI can still ignore advisory guard | Documented as "advisory first line"; security-reviewer catches violations |
| project-map.md in instructions adds 60-100 lines of context per session | **Minor** — premium context cost | 100-line hard cap; truncated to Hot Files + Constraints beyond that |
